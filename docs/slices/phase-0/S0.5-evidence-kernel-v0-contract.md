## (1) ID + Title
SLICE-P0-005 — 定義 evidence kernel 的 **v0 TS 契約**：一個 append-only、hash-chained、signed 的 log interface（`AppendOnlyLog`）+ 一個**獨立 verifier skeleton**（`verifyChain`），先在 TS plane 把不變量與測試立起來，作為 P1 Go kernel 的型別化契約來源。

## (2) Goal（一句話）
新增 `AppendOnlyLog`（只 `append`、回單調遞增 sequence + prev-hash 串接）與 `verifyChain(entries, publicKey)`（重算 hash-chain + 驗簽 + 偵測 gap/亂序/篡改），並以對抗式測試鎖定「篡改/亂序/缺序即 verifier FAIL」。

## (3) In-scope / Out-of-scope
- In-scope:
  - `src/audit/kernel/log.ts`：`AppendOnlyLog` interface — `append(entry): AppendReceipt`（攜 `sequence`、`contentHash`、`prevHash`、`entryHash`）；**無** `update`/`delete` 方法（append-only by construction，型別層即無改寫面）。
  - 一個 in-memory 參考實作 `InMemoryAppendOnlyLog`（僅供契約測試；非 production，無持久化）。
  - `src/audit/kernel/verify.ts`：`verifyChain(entries, publicKey): VerifyResult`（重算 prev-hash 串接、驗 Ed25519 checkpoint 簽章、回 `ok` 或第一個破點 `{ broken: true, at: sequence, reason }`）。
  - 以 `node:crypto` 的 Ed25519（零新依賴）做 checkpoint 簽章；leaf = SLICE-P0-002 的 `contentAddress`。
  - 對抗式測試：tamper、reorder、sequence gap、bad signature 皆使 `verifyChain` 回 broken。
- Out-of-scope（明確不做）:
  - 真實 Go 進程 / Tessera tile log / RFC-3161 外部錨定 / WASM verifier build → **P1**（本 slice 只立契約 + skeleton，刻意不跨進 Go plane，符合「P0 不重寫進 Go」）。
  - transactional outbox / 跨進程 gRPC ingest / monotonic per-source 多來源協調 → **P1**（本 slice 的 sequence 單調性是**單一 log 內**的，多來源協調留 P1）。
  - 控制平面與 kernel 的「不同進程/身分」結構隔離 → P1（本 slice 是同 process 內的契約，**不**宣稱已達成不可改寫的進程隔離；見 §6 對抗面註記）。

## (4) Design delta + modules + public interface + dependency direction
- **Design delta**: 目前 audit 只到「產生事件 + redact + serialize」。本 slice 立**證據鏈契約**：事件 leaf（content-address，S0.2）→ 串成 prev-hash chain → checkpoint 簽章 → 獨立 verifier 可重算驗證。型別層即排除改寫（無 update/delete），把 architecture-approach.md「control plane 無改寫權」在 TS 契約層先以「無改寫 API」近似（真正的進程/身分隔離在 P1）。
- **Modules touched（每個一句唯一責任，high cohesion 自證）**:
  - `src/audit/kernel/log.ts`（新增）— **唯一責任**：定義「只可 append、回 hash-chained receipt」的 log 契約與 in-memory 參考實作。
  - `src/audit/kernel/verify.ts`（新增）— **唯一責任**：獨立重算並驗證 chain 完整性（hash + sig + sequence），不依賴 log 的內部狀態（只看 entries）。
- **PUBLIC interface（新增/變更的對外公共面）**:
  - `export interface AppendOnlyLog { append(event: AuditEvent): AppendReceipt }`（**無**改寫方法）
  - `export interface AppendReceipt { sequence: number; contentHash: string; prevHash: string; entryHash: string }`
  - `export function verifyChain(entries: readonly LogEntry[], publicKey: KeyLike): VerifyResult`
  - `export type VerifyResult = { ok: true; length: number } | { ok: false; brokenAt: number; reason: string }`
  - `export class InMemoryAppendOnlyLog implements AppendOnlyLog`（標註「reference only, not durable」）
- **Dependency direction（low coupling 自證；HARD CONSTRAINT A）**:
  - 箭頭圖（向內、無 cycle）:
    ```
    src/audit/kernel ──▶ src/audit/canonical (contentAddress)  [S0.2]
                     ──▶ src/audit/event (AuditEvent)          [S0.1]
                     ──▶ node:crypto (Ed25519, sha256)
    ```
    （kernel 在 audit module 內、依賴向內 / 平台 stdlib；**verifier 不 import log 內部**——只吃 entries，使「獨立 verifier」名實相符。）
  - 僅經 public surface 消費（無 deep import）: ☑ 是。
  - 新依賴宣告: **無新第三方依賴**（`node:crypto` Ed25519）。
- **P1-conformance 契約常量（在此釘死，否則 P1 Go verifier 無法 byte-for-byte 對齊）：**
  - **genesis prevHash：** 第一筆（sequence 0）的 `prevHash` = 固定常量 `"sha256:" + 64 個 "0"`
    （即 `sha256:0000…0000`，32 bytes 全零的 hex），**非空字串、非省略欄位**。
  - **entryHash 定義：** `entryHash = contentAddress( canonicalBytes(event) ‖ prevHash ‖ sequence )`，
    其中 `‖` 為 §S0.2 canonical 框架下的確定性串接（長度前綴、無分隔歧義），演算法前綴沿用 `sha256:`（P1 可版本化到 `blake3:`）。
  - **checkpoint 簽章範圍：** Ed25519 簽的是 **whole-chain head**（對「最新 leaf 的 `entryHash` ‖ `length`」簽，
    即 checkpoint over chain head），**非 per-entry 逐筆簽**——這對齊 architecture-approach 的 Tessera checkpoint 模型，
    使 P1 Go 端可只簽 checkpoint 而非每筆。`verifyChain` 重算整鏈後，只驗這一個 checkpoint 簽章。

## (5) Test-first plan（先寫的 RED 測試）
- 測試檔: `src/audit/kernel/log.test.ts` + `src/audit/kernel/verify.test.ts`（新增）
- RED 測試清單（每條對應一個不變量）:
  - [ ] **Append-only / monotonic**：連續 append 三筆 → sequence 0,1,2；每筆 `prevHash` == 前一筆 `entryHash`（首筆 prevHash 為 genesis 常量）。
  - [ ] **Chain verifies**：完整鏈 + 正確 publicKey → `verifyChain` 回 `{ ok: true }`。
  - [ ] 安全對抗式（tamper）: 改動中間一筆 event 內容 → `verifyChain` 回 `{ ok: false, brokenAt: <該 seq> }`。
  - [ ] 安全對抗式（reorder）: 對調兩筆 entries 順序 → `verifyChain` broken。
  - [ ] 安全對抗式（gap）: 移除中間一筆（sequence 1,2,4）→ `verifyChain` broken（gap detection）。
  - [ ] 安全對抗式（bad sig）: 用錯 publicKey 驗 checkpoint → broken。
  - [ ] 安全對抗式（no rewrite surface）: 型別層確認 `AppendOnlyLog` **無** update/delete（編譯期；reviewer 嘗試呼叫不存在的方法應 `tsc` 失敗）。
