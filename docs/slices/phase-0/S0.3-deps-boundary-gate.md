- **Branch**: slice/p0-003-deps-boundary-gate
- **Author**: <id>    **Adversarial reviewer**: <須為 fresh-context、非作者>
- **Size budget**: 估計 <= 1 day；預期 net LOC <~110、files <~4、modules <~1（build/CI gate config）。**新增第三方依賴 `dependency-cruiser`（dev）—— 依 slice-spec §3「新依賴單獨成一個 slice」，本 slice 即是該唯一依賴變更，不混入行為變更。**

## (1) ID + Title
SLICE-P0-003 — 引入 `dependency-cruiser`、寫一條會抓到「cyclic / deep import / 反向依賴」的 RED 規則，並把 `deps:check` 接進 `pnpm run verify`，使 HARD CONSTRAINT A 由**指令 fail**強制、不再靠人眼。

## (2) Goal（一句話）
在 `pnpm run verify` 之中新增 `deps:check`（`depcruise src --config .dependency-cruiser.cjs`），其規則集對任何 cyclic dependency / 跨 module deep import / 反向（adapter→…→domain 以外的）依賴**以 error 級退非零**。

## (3) In-scope / Out-of-scope
- In-scope:
  - 新增 dev 依賴 `dependency-cruiser`（**本 slice 唯一的依賴變更**）。
  - `.dependency-cruiser.cjs`：至少 **error 級**規則 `no-circular`、`not-to-internal`（禁 deep import 跨 module 內部，只允 barrel）、`inward-only`（方向：adapters → application → domain，反向 error）。
  - `package.json` scripts 新增 `"deps:check": "depcruise src --config .dependency-cruiser.cjs"`，並把它**插入** `verify` 的 `test` 之後、`secret-scan` 之前（與 slice-spec §5 範例一致）。
  - 一個**故意違規的 fixture**（RED）證明規則真的會抓——驗證後移除或以 `// @deps-check-fixture` 隔離（見 §5）。
- Out-of-scope（明確不做）:
  - Python `import-linter`、Go `depguard`（那兩 plane 在 P1+ 才有真實程式碼）→ 留給 P1（kernel）/ 對應 SDK slice。
  - `eslint-plugin-boundaries`（替代方案）→ 不採用，避免雙工具重疊（YAGNI）。
  - 既有 src 的任何重構以「迎合」規則 → 本 slice 預期既有 scaffold **已合規**（audit→iam 單向）；若意外抓到既有違規，**該違規修正另開 slice**，本 slice 只接 gate。

## (4) Design delta + modules + public interface + dependency direction
- **Design delta**: `verify` 由 `typecheck && lint && build && test && secret-scan` 變為加入 `deps:check`：
  `typecheck && lint && build && test && deps:check && secret-scan`。新增一個 config 檔表達 layer 邊界。**這是把 slice-spec §4 的四點宣告從「人工自證」升級為「指令強制」的關鍵 slice。**
- **Modules touched（每個一句唯一責任，high cohesion 自證）**:
  - `package.json`（scripts）— **唯一責任**：宣告統一 gate 的子關卡組成。
  - `.dependency-cruiser.cjs`（新增）— **唯一責任**：以可執行規則表達「無環、無 deep import、向內」三條邊界律。
- **PUBLIC interface（新增/變更的對外公共面）**:
  - 命令面（contract）：`pnpm run deps:check`（exit 0 = 合規；exit≠0 = 有 cyclic/deep-import/反向依賴）；`pnpm run verify` 現含此關卡。
  - 無 TS API 變更。
- **Dependency direction（low coupling 自證；HARD CONSTRAINT A）**:
  - 本 slice **不**新增 module-to-module 依賴；它新增的是**對既有依賴圖的約束檢查**。
  - 新依賴宣告: `dependency-cruiser`（devDependency）—— 方向=僅 build-time 工具、不進 runtime 依賴圖；cycle=無（工具本身不被 src import）；理由=**operationalizes HARD CONSTRAINT A**，無替代的指令級執法手段，符合「不證明就不宣稱低耦合」。

## (5) Test-first plan（先寫的 RED 測試 — 此處「測試」= 故意違規 fixture 觸發規則）
> 本 slice 的 RED 不是 vitest 斷言，而是**讓 `deps:check` 對一個刻意植入的違規 fail**（RED→GREEN：先證明規則會抓，再讓 src 合規 → 綠）。
- RED 步驟（每條對應一條規則）:
  - [ ] 暫時加 `src/audit/_depfixture.ts`，內含 `import "../policy/evaluate.js"` 之外**再** deep import 一個內部路徑（模擬 `not-to-internal` 違規）或製造 `audit ↔ policy` 互引（模擬 `no-circular`）。
  - [ ] 跑 `pnpm run deps:check` 期望 **exit≠0** 並列出該違規（證明規則生效）。
  - [ ] 移除 fixture，跑 `pnpm run deps:check` 期望 **exit 0**（既有 scaffold 合規）。
- 首次紅燈證據（貼 exit≠0）:
  ```
  $ pnpm run deps:check        # with deliberate violation fixture present
  error no-circular: src/audit/_depfixture.ts → src/policy/evaluate.ts → ...
  ... 1 dependency violation (1 error, 0 warn)
  exit code: 1
  ```
- 額外（防回歸）：在 CI 上把「`deps:check` 必須對已知違規 fail」作為 S0.3 自身的 acceptance，避免規則被寫成永真。
