## (1) ID + Title
SLICE-P0-006 — 在**不弱化 deny-by-default / fail-closed** 的前提下，把 `evaluatePolicy` 擴充為支援**明確 deny 規則 + allow 規則**並建立 **deny-precedence 不變量**（任一 matching deny 永遠勝過 allow），作為 PDP layer 的種子。

## (2) Goal（一句話）
新增 `DenyRule` 與一個 `evaluatePolicy(input, { allow, deny })` 形態，使「有任何 matching deny → deny」「無 matching allow → deny」「僅在 matching allow 且無 matching deny 時 → allow」三條不變量由測試鎖定。

## (3) In-scope / Out-of-scope
- In-scope:
  - `src/policy/types.ts` 新增 `DenyRule`（與 `AllowRule` 同形：`id/action/resource`）+ `PolicyRuleSet`（`{ allow: AllowRule[]; deny: DenyRule[] }`）。
  - `evaluatePolicy` **向後相容地**接受新形態：既有 `(input, AllowRule[])` 維持可用；新增 `(input, PolicyRuleSet)`（deny 先評估、deny-precedence）。
  - 不變量：deny precedence、deny-by-default、fail-closed（malformed/throw→deny）全部保留並由測試覆蓋。
  - **wildcard-only deny 語義（在此拍板，非留待測試決定——這是安全決策不是測試細節）：**
    **deny 規則允許 wildcard-only pattern（`*`/`**`），且其語義為「over-match（寧可過度匹配也不漏 deny）」。**
    對稱地，**allow 規則的 wildcard-only pattern 仍維持既有 `evaluate.ts` 的「過寬→不視為有效 allow」處理**
    （即 wildcard-only allow 不放行）。兩者合起來保證：wildcard-only deny **絕不可能**反而「放行」任何請求；
    最壞情況只會 deny 更多（fail-safe 方向）。**明確排除** allow-override，deny precedence 對 wildcard-only
    deny 同樣成立。
- Out-of-scope（明確不做）:
  - OPA / Z3 整合（PEP 側）→ 留給 **P3**。
  - capability algebra（child⊆parent）、SoD conflict-pair、blast-radius budget、inference-route gate → 留給 **P4/P5**（這些是 killer-app 差異化邏輯，不在 P0 seed）。
  - 把 PolicyRequest 改用 SLICE-P0-001 的 AgentContext → 可選的後續小 slice；本 slice 不混入以保持單一意圖。

## (4) Design delta + modules + public interface + dependency direction
- **Design delta**: 目前 `evaluatePolicy(input, rules: AllowRule[])` 只有 allow 規則 + deny-by-default。新增**明確 deny 規則**與 deny-precedence：先掃 deny（任一 match → deny），再掃 allow（match → allow），皆無 → deny-by-default。這是 PDP 的最小擴充，**不**引入 allow-override（避免 deny 被繞過）。fail-closed 路徑（malformed input、評估 throw）原樣保留。
- **Modules touched（每個一句唯一責任，high cohesion 自證）**:
  - `src/policy/types.ts` — **唯一責任**：定義 policy 請求/規則/決策的型別（新增 DenyRule / PolicyRuleSet）。
  - `src/policy/evaluate.ts` — **唯一責任**：以 deny-by-default + deny-precedence + fail-closed 評估一個（可能 untrusted 的）請求。
- **PUBLIC interface（新增/變更的對外公共面）**:
  - `export interface DenyRule { id: string; action: string; resource: string }`
  - `export interface PolicyRuleSet { allow: readonly AllowRule[]; deny: readonly DenyRule[] }`
  - `evaluatePolicy` 多載：`evaluatePolicy(input: unknown, rules: readonly AllowRule[]): PolicyDecision`（既有）＋ `evaluatePolicy(input: unknown, rules: PolicyRuleSet): PolicyDecision`（新增）。回傳 `PolicyDecision`（含 `matchedRule`，deny 命中時亦回 matched deny rule id）。
- **Dependency direction（low coupling 自證；HARD CONSTRAINT A）**:
  - 箭頭圖（向內、無 cycle）:
    ```
    src/policy/evaluate ──▶ src/policy/types ──▶ src/iam (ids)
    ```
    （皆 policy module 內 + 向內到 iam；無新跨 module 邊、無 cycle。policy **不**依賴 audit/adapter——決策向下流、事件向上流，方向不可反。）
  - 僅經 public surface 消費（無 deep import）: ☑ 是。
  - 新依賴宣告: **無新第三方依賴**（沿用 zod）。

## (5) Test-first plan（先寫的 RED 測試）
- 測試檔: 擴充 `src/policy/evaluate.test.ts`
- RED 測試清單（每條對應一個不變量）:
  - [ ] **Deny precedence**：請求同時 match 一條 allow 與一條 deny → 結果 `deny`，`matchedRule` 為該 deny id。
  - [ ] **Allow only when clean**：match allow 且**無** matching deny → `allow`。
  - [ ] **Deny-by-default 保留**：空 ruleset（`{ allow: [], deny: [] }`）→ `deny`，reason 含 `deny-by-default`。
  - [ ] **向後相容**：既有 `(input, AllowRule[])` 呼叫行為**不變**（既有測試全綠 = 無回歸）。
  - [ ] 安全對抗式（fail-closed）: malformed input / ruleset 內含 null rule 觸發 throw → `deny`（reason 含 `fail-closed`），絕不 allow。
  - [ ] 安全對抗式（deny-by-default bypass）: deny 規則用 wildcard-only pattern（`*`/`**`）→ 依 §3 已拍板語義驗證：wildcard-only **deny over-match**（deny 更多）、wildcard-only **allow 不放行**；確認**絕不**因 deny pattern 而誤放行任何請求（fail-safe 方向）。
- 首次紅燈證據（貼 exit≠0；以 `PolicyRuleSet` 多載尚未存在）:
